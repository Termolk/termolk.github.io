4.8 Создание схемы базы данных

Правильно спроектированная база данных обеспечивает приложение актуальной информацией, упрощает его написание, поддержку и влияет на быстродействие. Поскольку правильный дизайн необходим, имеет смысл потратить время для создания хорошей схемы данных. А еще, показав свой проект базы данных старшим товарищам, вы сможете получить дельные советы до начала разработки!

В этом топике вы изучите

  • Некоторые термины базы данных.
  • Процесс проектирования.
  • Инструменты визуализации.

Частично основано на статье от Microsoft .

Термины

База данных организует вашу информацию в виде таблиц: списков, строк и столбцов, напоминающих блокнот или электронную таблицу.

Таблицы и сущности

В простой базе данных у вас может быть всего одна таблица (например, покупки в спиcке покупок). Для большинства баз данных потребуется больше. Например, у вас может быть таблица с информацией о товарах, таблица с информацией о заказах и таблица с информацией о покупателях. Товары и заказы иногда называют сущностями.

Записи и поля

Каждая строка таблицы называется записью, а каждый столбец - полем. Запись - это значимый и последовательный способ объединения информации о чем-либо. Поле - это отдельный элемент информации - тип элемента, который появляется в каждой записи. Например, в таблице «Продукты» каждая строка или запись будет содержать информацию об одном продукте. Каждый столбец или поле содержит информацию определенного типа об этом продукте, например его название или цену.

Процесс проектирования

Процесс проектирования устроен примерно так:

1. Определите цель вашей базы данных

Например, наша база предназначена для системы управления заказами (или заявками на курсы). Ее цель – хранить заказы, отдавать их менеджерам и обновлять данные заказов.

2. Соберите все типы информации. Разделите на таблицы.

Например, если вы пишете систему управления заказами, у вас скорее всего будут такие сущности, как "продукты", "заказы", "сотрудники", "туры", "сообщения".

3. Превратите информационные элементы в столбцы

Решите, какую информацию вы хотите хранить в каждой таблице. Каждый элемент становится полем и отображается в виде столбца в таблице.

Например, таблица «Сотрудники» может включать поля «фамилия» и «дата найма».

Например, таблица «Сообщения» может включать поля «отправитель», «получатель», «текст сообщения».

4. Укажите первичные ключи

Выберите первичный ключ каждой таблицы. Первичный ключ - это столбец, который используется для уникальной идентификации каждой строки.

Например, ID продукта или ID заказа.

5. Установите отношения между таблицами

Посмотрите на каждую таблицу и решите, как данные в одной таблице связаны с данными в других таблицах. При необходимости добавьте поля в таблицы или создайте новые таблицы, чтобы уточнить отношения.

Например, таблица «сообщения» связана с «пользователями», так как у каждого сообщения есть отправитель.Например, таблица «заказы» связана с «клиентами», так как у каждого заказа есть заказчик.

6. Уточните свой дизайн

Проанализируйте свой дизайн на наличие ошибок. Создайте таблицы и добавьте несколько записей образцов данных. Посмотрите, сможете ли вы получить желаемые результаты из ваших таблиц. Внесите коррективы в дизайн, по мере необходимости.

Например, после создания первых данных вы заметили, что в таблице «сообщения» не хватает «прочитанности» сообщения, а в таблице «пользователи» - аватарки. Добавьте их на этом шаге!

Правила выбора названий таблиц и атрибутов

Наименования таблиц и моделей

Название таблицы соответствует тому, что лежит в одном ее ряду. Например, для таблицы с преподавателями в одном ее ряду лежит преподаватель (teacher), поэтому таблица может называться teacher.

Класс модели в алхимии, который ее описывает, должен называться Teacher:

class Teacher(db.Model):

   __tablename__ = "teacher"
  id = db.Column(db.Integer(primary_key=True))

Если вы старинный поклонник SQL и предпочитаете, чтобы названия таблиц всегда шли во множественном числе, укажите это при определении названия таблицы:

class Teacher(db.Model):
   __tablename__ = "teachers"
  id = db.Column(db.Integer(primary_key=True))

Наименования атрибутов

Названия атрибутов идут в snake_case https://en.wikipedia.org/wiki/Snake_case. Если один из атрибутов это Foreign key, например, для таблицы object, то атрибут для него называется object_id.

Если у нас есть модели преподавателя и урока и у каждого урока есть один учитель, но у каждого учителя несколько уроков, это должно быть понятно из названий.

В модели Lesson будет:

class Lesson(db.Model):
    ...
    teacher_id = db.Column(db.Integer, db.ForeignKey("teacher.id"))
    teacher = db.relationship("Teacher")  # без S

В модели Teacher  будет:

class Teacher(db.Model):
    ...
    lessons = db.relationship("Lesson") # c S

Прежде, чем назвать поле, дважды спросите себя, в единственном или множественном числе будет его содержимое.

Схема базы данных

Хотя текстового описания моделей обычно достаточно, хорошая практика – визуализировать схему базы данных в виде диаграммы. Это особенно полезно, пока вы не начали "думать таблицами".

Во-первых, это делает базу более наглядной для вас.

Во-вторых, так вы можете показать промежуточный вариант коллегам или наставнику и быстро внести правки еще до того, как таблицы описаны и данные загружены.

Наконец, база данных выглядит не такой страшной, когда помещается на трети экрана. :)

Рисуем схему

Пора нарисовать схему.

Для примера спроектируем схему для мессенджера.

Зарегистрируйтесь на dbdesigner.net .

Создайте новую базу и добавляйте таблицы с помощью Insert>table .

1. Определяем цель

Цель нашей базы – хранить сообщения пользователей, чтобы показывать их в диалогах.

У каждого сообщения будет текст и отправитель, нам точно пригодятся пользователи.

Значит у нас будут Сообщения и Пользователи

2. Создаем таблицы

У нас будет по таблице для каждой сущности, Выбираем имена для таблиц:

– messages для сообщений,– users для пользователей.

3. Создаем поля и определяем их типы

У сообщения нас интересует:

– отправитель,

– получатель,

– текст сообщения,

– время, когда оно отправлено.

У пользователя нас интересуют:

– имя,

– фамилия.

Каждому полю зададим нужный подходящий тип данных.

Каждой таблице добавим id – будущий первичный ключ.

Можно было бы добавить еще данных, но для начала хватит и этих.

4. Создаем отношения между таблицами

Если подумать, информацию о пользователе непрактично хранить в таблице с сообщениями.

Записи могут быть такими:

idsenderreceivertextdatetime
78Лазарь ФедоровСерафим АндреевичДобрый день!12:23
79Лазарь ФедоровСерафим АндреевичУспели ли вы почитать наши отчеты?12:23
80Серафим АндреевЛазарь ФедоровичИ вам добрый!12:56
81Серафим АндреевЛазарь ФедоровичДа, как раз обсуждаем с коллегами!12:56

Как видно, хранение в полном виде отправителя и получателя - не самая лучшая идея.

Поэтому мы будем хранить не имена и фамилии отправителя и получателя, а ссылки на их строки в таблице пользователей.

Например, наша таблица пользователей выглядит вот так:

idnamesurname
25СерафимАндреев
33ЛазарьФедоров

Теперь перепишем таблицу сообщений, используя вместо имен id из таблицы users:

idsenderreceivertextdatetime
783325Добрый день!12:23
793325Успели ли вы почитать наши отчеты?12:23
802533И вам добрый!12:56
812533Да, как раз обсуждаем с коллегами!12:56

Свяжем две таблицы отношениями:

Для этого нажмите на "карандашик", проставьте чекбокс "Foreign Key" и выберите таблицу и ее поле, c которым вы хотите связаться отношениями.

5. Определим первичные ключи

Для нашей таблицы это несложная задача, поскольку у нас есть только один первичный ключ в каждой таблице.
