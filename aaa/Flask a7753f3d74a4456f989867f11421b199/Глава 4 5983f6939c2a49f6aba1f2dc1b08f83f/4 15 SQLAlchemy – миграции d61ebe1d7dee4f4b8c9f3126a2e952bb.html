4.15 SQLAlchemy – миграции

До этого момента каждое изменение структуры БД мы делали простым пересозданием БД.

В реальном мире мы так делать не можем, так как нам важны данные которые уже в БД есть.

В этом топике мы научимся:

  • Устанавливать инструмент для миграции.
  • Создавать миграции.
  • Модифицировать структуру базы данных.

Модификация таблицы

Когда мы создаём структуру БД, то можем допускать ошибки или требования нашей программы могут измениться. Изменить структуру можно или созданием новой базы, или модификацией существующей.

Мы можем:

  • Добавлять столбцы.
  • Удалять столбцы.
  • Добавлять ограничения.
  • Удалять ограничения.
  • Изменять значения по умолчанию.
  • Изменять типы столбцов.
  • Переименовывать столбцы.
  • Переименовывать таблицы.

Все эти действия - это формы команды ALTER TABLE.

Миграции

Миграции — это способ описания изменений в БД. Миграция, по сути, это файл с информацией об изменении. Их удобно хранить в системе контроля версий. С ними удобно строить инкрементальную, обратимую систему изменений структуры БД.

В SQLAlchemy за миграции отвечает пакет Alembic. Удобно настроить Alembic на работу с Flask можно с помощью дополнения Flask-Migrate. По факту в Alembic миграция - это файл с кодом на Python, который при миграции исполняется и генерирует SQL команды ALTER TABLE.

Flask-Migrate

Установим Flask-Migrate.

Для этого запустим команду pip install Flask-Migrate.

Так же как и с Flask-SQLAlchemy, нам нужно связать дополнение с объектом приложения, и передать ему ещё и объект Flask-SQLAlchemy.

from flask import Flask
from flask_sqlalchemy import SQLAlchemy
from flask_migrate import Migrate

app = Flask(__name__)
app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///test.db"
db = SQLAlchemy(app)
migrate = Migrate(app, db)

После этого запустим команду flask db init.  Эту, и все остальные команды flask db имя_команды нужно вводить в терминале (в консоли). Там же где и pip install

Эта команда создаст в проекте директорию migrations, в которой вся необходимая для миграций информация. В ней же находится директория versions, в которой в дальнейшем будут содержаться файлы наших миграций. Важно: команду flask db init нужно исполнять только один раз, не больше. Иначе есть шанс того, что уже готовые миграции повредятся.

Расценивайте директорию migrations как ваш код, держите её под контролем версий.

Первая миграция

Есть два способа создания миграций. Первый способ - это написать её руками. Второй способ - это положиться на автогенерацию миграций с помощью Alembic. Для этого напишите код с моделью и запустите команду flask db migrate.

Для примера код приложения:

from flask import Flask
from flask_migrate import Migrate
from flask_sqlalchemy import SQLAlchemy

app = Flask(__name__)
app.config["SQLALCHEMY_DATABASE_URI"] = "sqlite:///test.db"db = SQLAlchemy(app)
migrate = Migrate(app, db)


class User(db.Model):
    __tablename__ = "user"

    id = db.Column(db.Integer, primary_key=True)
    email = db.Column(db.String(255), unique=True, nullable=False)

После запуска команды flask db migrate в директории migrations/versions появился новый файл. Название у него будет разным у каждого из нас, так как оно генерируется случайным образом. При этом содержимое будет примерно совпадать.

Как видим, файл миграции - это файл с кодом на Python, который содержит информацию о том, какая это миграция. Например, revision это идентификатор ревизии, состояния БД после того, как миграция будет исполнена. Alembic контролирует переход из одной ревизии (состояния БД) в другую.

Текущую ревизию Alembic получает из специальной системной таблицы в БД, которую сам создаст при исполнении первой миграции (в коде миграции этой операции нет, так как она служебная).

down_revision - это идентификатор прошлой миграции, а так как у нас не было прошлых миграций то None.

Именно благодаря revision  и down_revision Alembic может генерировать изменения.

Ключевое в миграции - это функции upgrade и downgrade. Их вызывает Alembic в процессе применения или отката миграции соответственно.

Применение миграции

Для применения миграции выполните команду flask db upgrade.

Эта команда применит миграцию. Если все миграции уже применены, то не произойдёт ничего. Если более одной миграции не применено, то они исполнятся поочерёдно, в порядке указанном в down_revision. Зная изначальное состояние базы (первая миграция), и миграции после неё, Alembic высчитывает разницу между тем, что описывают модели и текущей структурой БД.

Откат миграции

Для того чтобы вернуть БД в состояние предыдущей миграции, выполните flask db downgrade.

Однако, откат миграции более сложная операция, потому есть шансы того, что она не удастся.

Автогенерация не идеальна

Важно! Alembic на текущий момент не может вычислить все изменения в моделях. Например, в данный момент не вычисляются изменения имени поля, констрейнта и таблицы. Автогенерация не предназначена для создания идеального файла миграции, рассматривайте её скорее, как удобный шаблон, который всё равно нужно прочитать и проверить прежде, чем исполнить.

Резюме

  • Изменения в структуру БД можно вносить с помощью команды ALTER TABLE.
  • Миграции это способ структурировано, инкрементально вносить изменения в структуру БД.
  • За миграции в SQLAlchemy отвечает пакет Alembic, а за связку Alembic и Flask — Flask-Migrate. Установить его можно командой pip install Flask-Migrate.
  • Инициализации Alembic через команду flask db init, она создаст необходимую директорию migrations.
  • Миграция в Alembic - это файл с кодом на Python.
  • Alembic может сам создавать файл с миграцией, за это отвечает команда flask db migrate.
  • За применение миграций отвечает команда flask db upgrade.
  • За отмену изменений, откат на версию назад отвечает команда flask db downgrade.
