5.8 Flask: структура приложения

Организация приложения Flask

Организация приложения — это технический подход к решению психологических ограничений и проблем разработчика. Любой человек может удерживать в уме ограниченное количество объектов. При этом мы, люди, плохо переключаемся между разными контекстами (медленно выгружаем из ума одни объекты и загружаем другие).

При этом нам, как разработчикам, нужно быстро вносить изменения в разрастающийся до сотен и тысяч строк код, не ломая сделанное. Как с этим разобраться? Хорошее решение – разобрать приложение на части и работать с каждой из них отдельно и по очереди.

Разделяй и властвуй — незаменимый подход в борьбе со сложностью, а, значит, и в разработке ПО.

Правильный разработчик не мечется между действиями (хаотичная смена контекста), а придерживается алгоритма для решения стоящих перед ним задач. Порой этот алгоритм может казаться избыточным, но он ведет к результату.

В этом топике мы изучим:

  • Как стоит разделять приложение Flask;
  • Почему это разделение - организация;
  • Модульная организация приложения;
  • Пакетная организация приложения;
  • Модульная разбивка пакета.

Пакетная организация

Разделение приложения на модули

Ранее код нашего приложения аутентификации и авторизации хранился в одном файле — модуле. Исключение: папка с шаблонами. Обычная структура модульного приложения содержит:

  • app.py — приложение;
  • config.py — настройки приложения;
  • requirements.txt — зависимости приложения для установки командой: pip install -r requirements.txt
  • static/ — папка для статичных файлов приложения (css, js, изображений и т.д.)
  • templates/ — папка шаблонов

Flask не против такой организации кода. Это нормально для маленького приложения. Маленького. А вот наше:

Оно уже большое, хотя не делает ничего полезного. Только как-то организует доступ. Наш код требует рефакторинг.

Рефакторинг - это не ругательство, а действия по улучшению качества кода без изменения его функционала. Существует целая классификация плохого кода, а вот цель одна — удобная организация для восприятия и внесения изменений. Ведь программирование — это больше правка кода.

После модуля, пакет — это вторая форма организации приложения Flask. Он содержит уже не один модуль, а их логически организованную группу.

Давайте из нашего модульного приложения сделаем пакет. Для этого выделим код в отдельные модули. И начнем мы с настроек.

Модуль настроек

Фреймворк Flask предоставляет единый механизм инициализации и хранения настроек через объект приложения. Кроме того, настройки можно загрузить из внешней структуры, например, как параметры класса. Создадим модуль с настройками:

Файл config.py

***

import os

# Для указания пути к файлу БД воспользумся путем до текущего модуля# - Текущая папка
current_path = os.path.dirname(os.path.realpath(__file__))
# - Путь к файлу БД в данной папке
db_path = "sqlite:///" + current_path + "\\test.db"

class Config:
    DEBUG = True
    SECRET_KEY= "secret_key"
    SQLALCHEMY_DATABASE_URI = db_path
    SQLALCHEMY_TRACK_MODIFICATIONS = False

Удалим настройки в модуле приложения. Импортируем класс настроек и применим:

Файл app.py

***

from config import Config

app = Flask(__name__)
app.config.from_object(Config)

Тестируем приложение, чтобы убедиться, что не сломали.

Размер кода сократился, но можно сделать ещё одно улучшение. Наш модуль приложения содержит и обработчики запросов, и инициализацию, и запуск приложения. Это как-то неправильно. Давайте выделим обработчики в отдельный модуль.

Модуль представлений (Views)

Так сложилось, что функции обработчики запросов, обычно называемые в других фреймворках контроллерами, во Flask принято называть представлениями (views). Поддержим традицию и назовем файл модуля "views.py". Вынесем в него декораторы и функции обработчики запросов (файл views.py):

Инициализация пакета

Остался один нерешенный вопрос. Наше приложение по-прежнему не является пакетом Python. Давайте исправим это.

Пусть папка, в которой находится приложение, называется "example1". Название папки является названием пакета. Для завершения создания пакета в папке приложения переименуем файл модуля приложения app.py в файл __init__.py. Также изменим обращения к модулям приложения с учетом названия пакета в модуле инициализации:

# Файл __init__.pyfrom flask import Flask

from example1.config import Config  #!!!from example1.models import db      #!!!

app = Flask(__name__)
app.config.from_object(Config)

db.init_app(app)

from example1.views import *        #!!!

Поправим импорт в модуле представлений:

# Файл views.py from functools import wraps

from flask import abort, flash, session, redirect, request, render_template

from example1 import app, db                                                #!!!from example1.models import User                                            #!!!from example1.forms import LoginForm, RegistrationForm, ChangePasswordForm  #!!!

(код модуля)

Структура нашего приложения в виде пакета выглядит так:

example1/
    templates/
    __init__.py
    config.py
    forms.py
    models.py
    views.py

Создадим скрипт запуска приложения run.py в папке, находящейся выше папки нашего пакета:

# Файл run.py import example1

example1.app.run()

Запустим скрипт из консоли. Работает!
