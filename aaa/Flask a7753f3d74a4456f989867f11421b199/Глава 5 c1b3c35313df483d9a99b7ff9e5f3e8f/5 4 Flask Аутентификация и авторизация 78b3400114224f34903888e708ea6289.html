5.4 Flask: Аутентификация и авторизация

В веб приложениях нам постоянно приходится отслеживать кто с ними работает. Социальная сеть, личный кабинет банка или онлайн-игра перед предоставлением доступа сначала идентифицируют пользователя и только потом предоставляют права. Так аутентификация и авторизация работают вместе.

Вы научитесь:

Различать аутентификацию и авторизацию.

Делать аутентификацию по логину и паролю.

Авторизовать идентифицированного пользователя на приватных страницах.

Теория

Чтобы разрешить или запретить пользователю открывать страницу или создавать сообщения нужны два механизма. Вот последовательность их работы:

  1. Пользователь неизвестен. У него минимальные права. Мы предлагаем ему представиться, например, ввести логин и пароль.
  1. Идентифицируем пользователя по предоставленным логину и паролю.
  1. Если нам известен такой пользователь, то выдаем пользователю на время взаимодействия уникальный идентификатор.
  1. При каждом новом запросе пользователь передает нам идентификатор, а система разрешает или запрещает доступ.

Первый механизм - идентификация пользователя - называется аутентификацией, а второй, отвечающий за выдачу прав на действия, авторизацией.

Часто,  например, на фриланс-биржах и при неформальном общении - используется только один термин. Дело в том, что слово "аутентификация" сложно для произношения и написания, а потому его могут пропустить и говорить об авторизации, когда требуются оба механизма. Не стоит сильно этим заморачиваться. Ещё одна причина для большей популярности термина авторизация в том, что авторизация работает часто сразу. Приложение на одной странице может предоставлять разный доступ для гостя и идентифицированного пользователя. Это и является работой авторизации.
Вообще, существует понятие трех больших А (AAA - Authentication, Authorization, Accounting - Аутентификация, Авторизация, Учет). Понятие  "учет" связано с учетом потребляемых пользователем ресурсов, например, доступное количество запросов к веб API, скачанных гигабайт или денег на игровом счете. В честь третьего понятия записи о каждом пользователе называют учетными записями (учетками) или аккаунтами.

В веб приложениях аутентификация осуществляется через форму логина, а авторизация - через механизм сессий, с которым вы ознакомились ранее.

Простая аутентификация по жестко установленным логину и паролю

Сейчас мы напишем веб приложение с простой авторизацией для одного пользователя. Информацию для доступа будем хранить в настройках Flask:

from flask import Flask

# Создаем приложение Flask
app = Flask(__name__)
# Настраиваем приложение Flask# - Секретный код для сессий
app.config["SECRET_KEY"] = "secret_key"# - Имя пользователя и пароль
app.config["USERNAME"] = "test"
app.config["PASSWORD"] = "test"
Прежде, чем мы продолжим, хотим предупредить: НИКОГДА не храните секретную информацию в настройках. Передавайте её либо через переменные окружения, либо храните в хранилище с ограниченным доступом, например, базе данных.

Пусть наша система будет состоять из двух страниц: секретной и публичной страницы аутентификации.

И давайте сразу ограничим доступ к приватной странице (сделаем авторизацию):

from flask import session, redirect, render_template

Авторизация готова! Но теперь нам нужно реализовать часть связанную с аутентификацией. Что нам понадобится? Нам понадобится форма и функция, обрабатывающая два запроса. Один запрос на отображение формы (GET запрос) и второй - реализующий аутентификацию (POST запрос).

Воспользуемся шаблонизатором для отображения формы (файл page_login.html):

<form action="/login" method="post">
  Имя: <input type="text" name="username" value="">
  Пароль: <input type="password" name="password"><button type="submit">Войти</button></form>

А так выглядит полный код авторизации:

from flask import Flask, session, redirect, request, render_template

# Создаем приложение Flask
app = Flask(__name__)
# Настраиваем приложение Flask# - Секретный код для сессий
app.config["SECRET_KEY"] = "secret_key"# - Имя пользователя и пароль
app.config["USERNAME"] = "test"
app.config["PASSWORD"] = "test"# Мы обрабатываем по данному URL и GET, и POST запрос@app.route("/login", methods=["GET", "POST"])def login():# Если пользователь авторизованif session.get("is_auth"):
        # то редиректим его на приватную страницуreturn redirect("/")

    # Переменная для хранения ошибок авторизации
    error_msg = ""  # Пока ошибок нет# Пришел запрос на аутентификациюif request.method == "POST":

        # Получаем отправленные данные
        username = request.form.get("username")
        password = request.form.get("password")

        # Проверяем полученные данныеif ((username and password) and username == app.config["USERNAME"] and password == app.config["PASSWORD"]):

            # Устанавливаем в сессии признак, что пользователь аутентифицирован
            session["is_auth"] = True# Редиректим пользователя на приватную страницуreturn redirect("/")

        else:

            error_msg = "Неверное имя или пароль"# Отображаем форму аутентификацииreturn render_template("page_login.html")

app.run(debug=True, host='0.0.0.0', port=8080)

Разлогинивание

Теперь у нас есть веб приложение с авторизацией! Но что-то мы все же забыли. Мы забыли добавить выход из приложения. Что нужно для выхода? Удалить из сессии параметр is_auth. Сделаем это мы на отдельном роуте:

@app.route('/logout/', methods=["POST"])
def logout():
    if session.get("is_auth"):
        session.pop("is_auth")
    return redirect("/login")

А на приватную страницу добавим форму выхода:

<form action="/logout/" method="post"><button type="submit">Выйти</button></form>

У нас получилось простое приложение с авторизацией. Оно не идеально, например:

  • Нам нужно вставлять код авторизации во все новые страницы, которые мы захотим добавить.
  • Верификация запроса аутентификации тоже не идеальна.
  • Мы работаем с одним пользователем.
  • Нового пользователя нельзя добавить в любой момент.
  • Наше приложение не безопасно. Пароль хранится в открытом виде в теле самого приложения.

С этими проблемами мы разберемся в следующих уроках.

Простая аутентификация из базы данных

Мы научились простой аутентификации и авторизации пользователя. Но что если нам нужно дать доступ к веб приложению нескольким пользователям? Для хранения их аккаунтов нам понадобится база данных (БД).

Совместное использование с WTF

Чтобы улучшить работу с формами мы не станем изобретать велосипед.

Опишем формы регистрации (файл app.py):

from flask_wtf import FlaskForm
from wtforms import StringField, PasswordField
from wtforms.validators import DataRequired, Length, EqualTo

# Форма аутентификацииclass LoginForm(FlaskForm):# Добавляем поле имени пользователя
    username = StringField("Имя:", validators=[DataRequired()])

    # Добавляем поле пароля
    password = PasswordField("Пароль:", validators=[DataRequired()])

Пакет Flask-WTF позволяет не только декларировать типы полей формы, назначать их проверку (validation) и отображать в шаблоне. Он также поддерживает защиту форм от CSRF атак. Это очень важно!

Изменим функцию аутентификации с использованием формы LoginForm (файл app.py):

@app.route("/login", methods=["GET", "POST"])
def login():
    if session.get("user_id"):
        return redirect("/")

    # Создаем форму
    form = LoginForm()

    if request.method == "POST":

        # Если форма не валиднаif not form.validate_on_submit():

            # показываем форму и не забываем передать форму в шаблон            return render_template("page_login.html", form=form)

        #Информацию о пользователе берем из базы по введенной почте
        user = User.query.filter(User.name == form.username.data).first()

        # Данные берем из формыif not user or user.password != form.password.data:

            # Добавляем ошибку для поля формы
            form.username.errors.append("Неверное имя или пароль")

        else:
            session["user_id"] = user.id            return redirect("/")

    return render_template("page_login.html", form=form)

Выше в коде видно, что мы убрали переменную error_msg и ошибки добавляем прямо в форму.Это сделало наш код более понятным.

Резюме

Аутентификация идентифицирует пользователя.

Авторизация выдает права на основе информации о пользователе.

Простая аутентификация осуществляется по логину и паролю. Логин идентифицирует какой именно пользователь обратился. Пароль же подтверждает, что пользователь именно тот.

Для хранения идентификационной информации в большинстве веб приложений используются сессии. На основе этой информации осуществляется авторизация.

Авторизация может быть основана на признаке аутентификации, по идентификатору пользователя и роли.
