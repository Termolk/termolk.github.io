5.9 Flask – Blueprints

До этого мы писали в основном файле app.py. Когда всё приложение написано в одном файле, оно быстро становится неконтролируемо большим. Появляется необходимость разделить приложение по логическим кусочкам. Для этого во фласке есть блупринты.

Вы научитесь:

– Писать большие приложения, разделяя логику по разным блупринтам.

– Создавать и использовать блупринты.

Модульность приложения

Ранее мы писали обработчик (view) явно указывая объект приложения.

from flask import Flask

app = Flask(__name__)


@app.route("/")
def hello_world():
	return "Hello, World!"

Тут мы глобально создали объект приложения, и явно зарегистрировали функцию обработчик.

Этот код отлично работает для небольшого приложения. Но нам всегда хочется модульности, возможности разделять код приложения, по модулям и пакетам.

Во Flask мы можем поступить двумя способами, один не исключает другого:

  • Разбить нашу логику по блупринам.
  • Разбить нашу логику по разным объектам приложения (app).

Что такое blueprint

Blueprint во Flask - это объект, который похож на приложение (app), но на самом деле не является приложением. Дословный перевод слова blueprint — копия строительного чертежа на светобумаге. Это хорошая аналогия тому, что именно делает blueprint во Flask.

Итак, блупринт это набор операций, имеющий имя, которые вместе регистрируются в приложении.

Зачем?

Блупринты позволяют:

  • Разделить приложение на набор блупринтов. Это здорово для большого приложения, когда проект может создать объект приложения и разные наборы блупринтов в зависимости от условий задачи.
  • Блупринты могут иметь свои отдельные шаблоны, фильтры, статические файлы. Блупринт может даже не реализовывать ни одной view.
  • Блупринты можно использовать для разделения по URL префиксам и/или поддоменам.
  • Блупринт может регистрироваться дополнениями (extension) при инициализации дополнения. Так, например, работает Flask-Admin, предоствляя свои шаблоны и логику, записанную в блупринт.

Давайте посмотрим на тестовом примере

Ссылка на репозиторий с тестовым примером.

У нас есть два блупринта, каждый отдельно в своей директории.

Первые блупринты

Давайте посмотрим на наш первый блупринт. В репозитории он находится в директории simple_page в файле views. Он имеет два роута.

from flask import Blueprint


simple_page = Blueprint("simple_page", __name__)


@simple_page.route("/<page>")
def show(page):
	return page * 3

@simple_page.route("")
def main():
	return "Hit simple main page"

Объект Blueprint очень похож на объект приложения. После создания мы можем определять на нём роуты, фильтры, иметь свои шаблоны и статические файлы.

Второй блупринт иллюстрирует более "сложную" работу.

from flask import Blueprint

hard_page = Blueprint("hard_page", __name__)


@hard_page.route("/<page>")
def show(page):
	return page * 300

@hard_page.route("")
def my_main():
	return "<b>Hit rich HTML main page</b>"

Для того чтобы блупринты заработали, в приложении их нужно зарегистрировать.

Регистрация блупринта

Блупринт записывает все операции которые нужно выполнить когда его зарегистрируют в приложении.

Как зарегистировать блупринт?


app = flask.Flask(__name__)
app.register_blueprint(simple_page, url_prefix="/simple")

После этого приложение получает всё что блупринт запомнил до этого, в нашем случае это два роута. url_prefix это не обязательный аргумент, указывает какой префикс будет у ссылок ведущих в наш блупринт.

Пакеты и модульность

Само по себе у нас возникает желание разложить разные блупринты по разным модулям и директориям. Сделать это достаточно просто, главное чтобы модуль был импортируемым. Лучше всего сразу раскладывать блупринты по разным пакетам.

Например, у нас есть код приложения такси.

models/
  car.py
api/
  __init__.py
  orders.py
my_site/
  __init__.py
  views.py
  templates/
    main.html
app.py

На примере нашего приложения.

hard_page/
  __init__.py
  views.py
simple_page/
  __init__.py
  views.py
app.py

Модуль app.py содержит код создания приложения и запуска сервера разработки.

Пакеты hard_page и simple_page будут содержать наши блупринты. В реальном приложении скорее всего эти пакеты содержали бы много разных файлов, логически составляющих единое целое. Например код схем моделей конкретного блупринта, свои шаблоны, статику и прочие вещи.

Шаблоны и вспомогательные функции

У блупринтов могут быть свои шаблоны, статические файлы, фильтры и другие вспомогательные функции. Блупринт даже не обязан имплементировать вьюхи.

Например вот так можно подключать свои шаблоны в блупринт. При этом они будут доступны только из блупринта, и будут иметь более низкий приоритет чем общие шаблоны приложения.

bp = flask.Blueprint('flights', __name__, template_folder='templates')

Пример файловой структуры:

flights/
  templates/
    main.html
  flight.py
app.py

Теперь рендер шаблона main.html сработает из нашего блупринта:

@bp.route("")
def main():
	return flask.render_template("main.html")

Резюме

Блупринты нужны когда мы хотим логически выделить разные части приложения в разные места.

Удобно их использовать для логической сепарации, например по поддоменам или префиксам.

Так же блупринты могут использоваться при написании дополнений (Flask extensions)

Регистрация блупринтов в главном файле приложения:


app = flask.Flask(__name__)
app.register_blueprint(simple_page, url_prefix="/simple")

Код описания представлений блупринта

from flask import Blueprint


simple_page = Blueprint("simple_page", __name__)


@simple_page.route("/<page>")
def show(page):
	return page * 3
